# 코드 분석 워크플로우

Claude Code를 활용한 체계적인 코드 분석 방법을 안내합니다. 레거시 코드 이해부터 보안 감사까지, 실무에서 바로 활용할 수 있는 프롬프트와 기대 결과를 정리했습니다.

---

## 1. 레거시 코드 이해

오래된 프로젝트나 처음 접하는 코드베이스를 빠르게 파악해야 할 때 사용합니다.

### 기본 프롬프트

```
이 프로젝트의 전체 구조를 설명해줘
```

### 심화 프롬프트 예시

```
이 프로젝트의 디렉토리 구조를 분석하고, 각 폴더의 역할과 주요 파일들의 관계를 설명해줘.
특히 진입점(entry point)이 어디인지, 데이터 흐름이 어떻게 되는지 다이어그램 형태로 보여줘.
```

```
src/services/ 디렉토리의 코드를 분석해줘. 각 서비스가 어떤 책임을 가지고 있고,
서로 어떻게 의존하는지 정리해줘.
```

```
이 프로젝트에서 사용하는 디자인 패턴들을 찾아서 설명해줘.
각 패턴이 어디에 적용되어 있고, 왜 그 패턴을 선택했는지 추론해줘.
```

### Claude의 기대 동작

- 프로젝트 루트의 `package.json`, `tsconfig.json`, `Makefile` 등 설정 파일을 먼저 확인
- 디렉토리 구조를 탐색하여 전체 레이아웃 파악
- 주요 진입점 파일을 읽고 의존성 그래프를 추적
- 계층별(컨트롤러, 서비스, 모델 등) 역할 분류 제시

### 실전 팁

> 프로젝트가 클 경우, 특정 디렉토리나 모듈을 지정하면 더 정확한 분석 결과를 얻을 수 있습니다.
> 예: `src/auth/ 모듈의 인증 흐름을 분석해줘`

## 2. 아키텍처 분석

시스템의 설계 패턴과 구조적 특성을 파악합니다.

### 기본 프롬프트

```
이 앱의 아키텍처 패턴을 분석해줘
```

### 심화 프롬프트 예시

```
이 프로젝트가 클린 아키텍처를 따르고 있는지 분석해줘.
각 레이어(도메인, 유스케이스, 인프라)가 제대로 분리되어 있는지,
의존성 방향이 올바른지 확인해줘.
```

```
이 마이크로서비스 간의 통신 패턴을 분석해줘.
동기/비동기 호출이 어디에 사용되고, 장애 전파 가능성이 있는 지점을 찾아줘.
```

```
이 프론트엔드 앱의 상태 관리 구조를 분석해줘.
전역 상태와 로컬 상태가 어떻게 분리되어 있고, 데이터 흐름이 단방향인지 확인해줘.
```

### Claude의 기대 동작

- 프로젝트의 레이어 구조(MVC, MVVM, 클린 아키텍처 등) 식별
- 모듈 간 의존성 방향 분석
- API 라우트 정의와 미들웨어 구조 파악
- 상태 관리 라이브러리와 패턴 식별
- 개선이 필요한 구조적 문제점 제시

### 실전 팁

> 아키텍처 분석 후 개선점을 물어보면 구체적인 리팩토링 방향을 제안받을 수 있습니다.
> 예: `분석 결과를 바탕으로 아키텍처 개선 방안을 우선순위와 함께 제안해줘`

## 3. 버그 원인 분석 (Root Cause Analysis)

에러 메시지나 비정상 동작의 근본 원인을 추적합니다.

### 기본 프롬프트

```
이 에러의 원인을 추적해줘
```

### 심화 프롬프트 예시

```
다음 에러가 프로덕션에서 발생하고 있어. 원인을 추적해줘:

TypeError: Cannot read properties of undefined (reading 'map')
    at UserList (src/components/UserList.tsx:23:18)
    at renderWithHooks (node_modules/react-dom/...)

이 컴포넌트에 전달되는 데이터 흐름을 따라가서 어디서 undefined가 발생하는지 찾아줘.
```

```
API 호출 시 간헐적으로 504 Gateway Timeout이 발생해.
관련 코드를 분석해서 타임아웃이 발생할 수 있는 모든 지점을 찾아줘.
특히 데이터베이스 쿼리와 외부 API 호출 부분을 중점적으로 봐줘.
```

```
다음 테스트가 CI에서만 실패하고 로컬에서는 통과해:

FAIL src/utils/date.test.ts
  Expected: "2024-03-15"
  Received: "2024-03-14"

타임존 관련 이슈인지 분석하고, 수정 방법을 제안해줘.
```

### Claude의 기대 동작

- 스택 트레이스에서 언급된 파일과 라인을 즉시 확인
- 해당 코드의 데이터 흐름을 역추적
- 관련 함수 호출 체인을 따라가며 원인 범위를 좁힘
- 가능한 원인을 우선순위로 나열
- 각 원인에 대한 수정 방안 제시

### 실전 팁

> 에러 메시지와 함께 재현 조건(특정 입력값, 환경 등)을 알려주면 분석 정확도가 크게 올라갑니다.
> 스택 트레이스 전체를 붙여넣는 것이 부분만 넣는 것보다 효과적입니다.

## 4. 의존성 분석

프로젝트의 의존성 상태를 점검하고 불필요한 패키지를 식별합니다.

### 기본 프롬프트

```
사용하지 않는 의존성을 찾아줘
```

### 심화 프롬프트 예시

```
package.json의 dependencies와 devDependencies를 분석해서:
1. 코드에서 실제로 import하지 않는 패키지
2. 다른 패키지와 기능이 중복되는 패키지
3. deprecated된 패키지
를 각각 찾아줘.
```

```
이 프로젝트의 의존성 트리를 분석해서 보안 취약점이 있는 패키지가 있는지 확인해줘.
대안 패키지도 함께 제안해줘.
```

```
번들 사이즈에 가장 큰 영향을 미치는 의존성 TOP 5를 찾고,
더 가벼운 대안이 있으면 제안해줘.
```

### Claude의 기대 동작

- `package.json` (또는 `requirements.txt`, `go.mod` 등)의 의존성 목록 확인
- 프로젝트 전체에서 각 패키지의 import/require 사용 여부 검색
- 사용되지 않는 의존성 목록 작성
- 중복 기능을 가진 패키지 식별
- 제거 시 영향 범위 분석

### 실전 팁

> 의존성 제거 전에 반드시 `해당 패키지를 제거하면 영향받는 코드를 모두 찾아줘`로 확인하세요.
> 간접 의존성(다른 패키지의 peer dependency)으로 사용되는 경우도 있습니다.

## 5. 보안 감사

코드의 보안 취약점을 체계적으로 검토합니다.

### 기본 프롬프트

```
보안 취약점을 검토해줘
```

### 심화 프롬프트 예시

```
이 프로젝트의 보안 취약점을 OWASP Top 10 기준으로 검토해줘.
특히 다음 항목을 중점적으로 확인해줘:
- SQL Injection 가능성
- XSS 취약점
- 인증/인가 우회 가능성
- 민감 정보 노출
```

```
src/api/ 디렉토리의 모든 엔드포인트를 검토해서:
1. 입력값 검증이 누락된 곳
2. 인증 미들웨어가 빠진 라우트
3. Rate limiting이 없는 민감 API
를 찾아줘.
```

```
환경변수와 시크릿 관리를 검토해줘.
하드코딩된 API 키나 비밀번호가 있는지,
.env 파일이 .gitignore에 포함되어 있는지,
민감 정보가 로그에 출력되는지 확인해줘.
```

```
이 인증 시스템의 보안을 검토해줘:
- JWT 토큰 처리 방식
- 비밀번호 해싱 알고리즘
- 세션 관리 로직
- CORS 설정
```

### Claude의 기대 동작

- 전체 코드베이스에서 보안 관련 패턴 검색 (SQL 쿼리, 사용자 입력 처리 등)
- 인증/인가 미들웨어 적용 여부 확인
- 환경변수 사용 패턴 검토
- 알려진 취약 패턴(eval, innerHTML 직접 사용 등) 탐지
- 취약점별 심각도와 수정 방법 제시

### 실전 팁

> 보안 감사는 범위를 좁혀서 반복 실행하는 것이 효과적입니다.
> 예: 먼저 `인증 관련 코드만 보안 검토해줘`, 그 다음 `API 입력값 검증을 검토해줘` 순서로 진행하세요.

## 6. 성능 분석

코드 수준의 성능 병목 지점을 식별합니다.

### 기본 프롬프트

```
성능 병목을 찾아줘
```

### 심화 프롬프트 예시

```
데이터베이스 관련 코드를 분석해서 성능 이슈를 찾아줘:
- N+1 쿼리 패턴
- 인덱스 없이 자주 조회되는 컬럼
- 불필요한 전체 테이블 스캔
- 트랜잭션이 너무 길게 유지되는 곳
```

```
이 React 앱의 렌더링 성능을 분석해줘:
- 불필요한 리렌더링이 발생하는 컴포넌트
- 메모이제이션이 필요한 곳
- 번들 스플리팅이 가능한 라우트
- 큰 리스트에서 가상 스크롤이 필요한 곳
```

```
이 API 서버의 응답 시간을 개선할 수 있는 부분을 찾아줘:
- 캐싱이 가능한 응답
- 병렬화할 수 있는 순차 호출
- 불필요한 데이터 변환
- 최적화할 수 있는 알고리즘
```

### Claude의 기대 동작

- 데이터베이스 쿼리 패턴 분석 (ORM 코드 포함)
- 반복문 내 비효율적 로직 탐지
- 캐싱 가능한 연산 식별
- 비동기 처리 최적화 가능 지점 탐색
- 구체적인 개선 코드와 예상 효과 제시

### 실전 팁

> 성능 분석 시 실제 병목 데이터(프로파일러 결과, 느린 쿼리 로그 등)가 있다면 함께 제공하세요.
> 추측 기반 최적화보다 데이터 기반 최적화가 훨씬 효과적입니다.

## 분석 워크플로우 조합 예시

여러 분석을 순차적으로 조합하면 프로젝트를 종합적으로 파악할 수 있습니다.

### 신규 프로젝트 온보딩

```
1단계: 이 프로젝트의 전체 구조를 설명해줘
2단계: 핵심 비즈니스 로직이 있는 모듈을 찾아서 데이터 흐름을 설명해줘
3단계: 테스트 커버리지가 낮은 핵심 모듈을 찾아줘
4단계: 기술 부채가 있는 부분을 우선순위와 함께 정리해줘
```

### 프로덕션 이슈 대응

```
1단계: 이 에러 로그를 분석해서 원인을 추적해줘: [에러 로그 붙여넣기]
2단계: 관련 코드의 최근 변경 이력을 확인해줘
3단계: 같은 유형의 에러가 발생할 수 있는 다른 코드도 찾아줘
4단계: 수정 코드와 함께 재발 방지를 위한 테스트 코드를 작성해줘
```

### 정기 코드 품질 점검

```
1단계: 보안 취약점을 검토해줘
2단계: 사용하지 않는 의존성과 코드를 찾아줘
3단계: 성능 병목을 찾아줘
4단계: 발견된 이슈를 심각도별로 정리하고 개선 로드맵을 제안해줘
```
